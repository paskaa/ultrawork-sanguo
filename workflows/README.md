# UltraWork 完整工作流指南

## 🔄 工作流 vs 单点调度

UltraWork 支持两种调度模式：

| 模式 | 命令 | 流程 | 适用场景 |
|------|------|------|----------|
| **单点调度** | `/ulw <任务>` | 直接指派一个主将 | 简单任务、快速修复 |
| **完整工作流** | `/workflow <类型> <任务>` | 6阶段完整流程 | 复杂任务、需要质量保证 |

## 📋 工作流类型

### 1. Bug修复工作流

```bash
/workflow bugfix 修复登录页面验证码不显示的问题
```

**流程**: 分析 → 规划 → 修复 → 审查 → 测试 → 监控

**执行团队**:
- 分析: 司马懿 + 司马师
- 规划: 周瑜 + 鲁肃
- 修复: 张飞 + 赵云
- 审查: 关羽 + 关平
- 测试: 徐庶 + 庞林
- 监控: 满宠 + 程昱 + 贾诩

---

### 2. 功能开发工作流

```bash
/workflow feature 实现用户个人中心页面
```

**流程**: 需求 → 架构 → 开发 → 审查 → 测试 → 部署

**执行团队**:
- 需求: 周瑜 + 鲁肃
- 架构: 周瑜 + 鲁肃

- 审查: 关羽 + 关平

---

### 3. 代码重构工作流

```bash
/workflow refactor 重构订单模块的代码结构
```

**流程**: 分析 → 规划 → 重构 → 审查 → 测试 → 监控

---

### 4. 代码审查工作流

```bash
/workflow review 审查用户管理模块的代码
```

**流程**: 静态分析 → 安全检查 → 质量评估 → 综合报告

---

### 5. 监控诊断工作流

```bash
/workflow monitor 诊断系统异常
```

**流程**: 前端监控 → 后端监控 → 日志分析 → 综合诊断

## 🔁 循环修复机制

如果任何阶段失败，工作流会自动循环：

```
[Phase 5: 测试] ❌ 测试失败
    ↓
[循环判断] 重试次数: 1/3
    ↓
返回 [Phase 3: 修复]
    ↓
重新执行修复 → 审查 → 测试
```

最多重试3次，超过则标记工作流失败。

- `workflows/refactor-workflow.md` - 代码重构

## 💡 使用建议

### 什么时候使用完整工作流？

**使用完整工作流 (workflow)**:
- Bug修复，特别是复杂Bug
- 新功能开发
- 代码重构
- 需要质量保证的任务
- 生产环境修复

**使用单点调度 (ulw)**:
- 快速小问题修复
- 简单查询
- 探索性任务
- 开发环境调试

### 选择合适的团队

```
前端问题 → /workflow bugfix 前端问题
后端问题 → /workflow bugfix 后端问题
新功能   → /workflow feature 功能描述
代码优化 → /workflow refactor 重构描述
安全审查 → /workflow review 审查范围
系统异常 → /workflow monitor 诊断
```
